# Архитектура сервиса

Документ описывает архитектуру сервиса прогнозирования дефолта по кредитным картам, обоснование выбранного подхода и концепты MLOps, применимые при развитии проекта.

## 1. Обзор

Сервис состоит из одного Flask-приложения, упакованного в Docker-контейнер. Приложение при старте загружает в память две версии модели (`model_v1.pkl`, `model_v2.pkl`), маршрутизация между версиями выполняется внутри сервиса.

В docker-compose перед сервисом стоит nginx как reverse proxy: внешний трафик идёт на порт 80 nginx, а порт 5000 ml-service не публикуется на хост.

```
+------------------------------------------------------------+
|  docker-compose                                            |
|                                                            |
|  +-------------+        +--------------------------------+ |
|  |   nginx     |  -->   |  ml-service (Flask)            | |
|  |  :80        |        |   /health                      | |
|  | reverse     |        |   /predict                     | |
|  | proxy       |        |     -> validate_input          | |
|  +-------------+        |     -> choose_version (A/B)    | |
|         ^               |     -> model.predict           | |
|         |               |                                | |
|   client (curl,         |   model_v1.pkl  model_v2.pkl   | |
|   frontend, batch)      |   (in-memory Pipeline)         | |
|                         |                                | |
|                         |   stdout: JSON-логи            | |
|                         +--------------------------------+ |
+------------------------------------------------------------+
```

Компоненты:
- `app/api.py` - Flask-приложение, два эндпоинта (`/health`, `/predict`)
- `app/model_handler.py` - валидация входа, выбор версии, инференс
- `models/model_v[12].pkl` - обученные `sklearn.pipeline.Pipeline` (StandardScaler + GradientBoostingClassifier)
- `docker/nginx.conf` - конфиг reverse proxy на `ml-service:5000`
- JSON-логи в stdout - контракт с production-сборщиком (Filebeat, Fluentd)

## 2. Монолит против микросервисов

Для проекта выбран **монолит**.

Аргументы за монолит в данном кейсе:
- Сервис решает одну задачу - возвращает прогноз дефолта по вектору признаков. Подсистем с независимыми циклами разработки нет
- Две версии модели (v1, v2) - это варианты внутри одного сервиса, а не разные продукты. Вынос каждой в отдельный контейнер усложнил бы маршрутизацию A/B и удвоил инфраструктуру без выигрыша
- Предобработка и инференс выполняются в одном процессе. Сетевых вызовов между шагами нет, latency минимальна
- Развёртывание из одного образа: один артефакт в Docker Hub, одна точка наблюдения, одна стратегия отката
- Горизонтальное масштабирование делается запуском нескольких реплик того же образа за балансировщиком - этого достаточно для предсказуемой нагрузки сервиса скоринга

Когда переход на микросервисы оправдан:
- Подсистемы требуют разного железа (CPU-предобработка и GPU-инференс)
- Разные команды владеют разными частями pipeline и хотят независимые релизы
- Профиль нагрузки компонентов сильно различается (например, тяжёлый ETL + лёгкий API)
- Появляются smart-features рядом с инференсом (feature store, online-aggregation), которые удобнее держать отдельно

В учебном сценарии ни одно из этих условий не выполняется, поэтому усложнение архитектуры на данном этапе избыточно.

## 3. Брокер сообщений (RabbitMQ)

В рамках проекта брокер не реализован. Описание даётся для понимания, как сервис расширяется при росте нагрузки.

Сценарии применения RabbitMQ:
- **Асинхронные батч-предсказания.** Запрос на скоринг файла с десятками тысяч заявок не помещается в синхронный HTTP. Клиент кладёт задачу в очередь, воркеры разбирают её пачками, результат пишется в БД, клиент получает уведомление о готовности
- **Сглаживание пиков нагрузки.** При всплеске количества HTTP-запросов в секунду (RPS) очередь буферизует входящие запросы, воркеры обрабатывают их с допустимой задержкой, ошибок 503 не возникает
- **Раздельное логирование.** Каждый запрос публикуется в топик "predictions", подписчики (analytics-ETL, сервис мониторинга качества модели) разбирают события независимо. Скорость инференса не зависит от скорости логирования

Для всех трёх сценариев нужно вынести инференс в отдельный воркер-процесс с подпиской на очередь.

## 4. Логирование и мониторинг

Каждый запрос пишется в stdout в JSON по одной строке. Поля: `timestamp`, `level`, `endpoint`, `method`, `status`, `model_version`, `prediction`, `probability`. Необработанные исключения логируются с `error_type`, `error_message` и срезом traceback.

Production-путь логов:
- stdout контейнера читает сборщик (Filebeat, Fluentd) - стандартный механизм для контейнеризованных сервисов
- сборщик отправляет события в Elasticsearch
- Kibana визуализирует RPS, распределение `probability`, долю 4xx/5xx, баланс трафика v1 и v2 в A/B-тесте

Связка ELK (Elasticsearch + Logstash + Kibana) - индустриальный стандарт для production-логов. В рамках проекта интеграция не реализуется, JSON-формат сохраняется как контракт.

## 5. DVC и MLflow

**DVC (Data Version Control)** - версионирование данных и моделей в Git-подобной парадигме. Применимо для:
- хранения `data/UCI_Credit_Card.csv` вне Git (для датасетов в гигабайтах Git непригоден)
- привязки версии `.pkl` к версии данных и параметрам обучения через `dvc.yaml`
- автоматического перезапуска обучения при изменении входов командой `dvc repro`

**MLflow** - трекинг экспериментов и реестр моделей. Применимо для:
- сохранения каждого прогона `train_model.py` с параметрами и метриками F1, Precision, Recall, ROC AUC
- сравнения v1 и v2 в едином UI
- регистрации моделей со стадиями `Staging`, `Production` и развёртывания через `mlflow models serve`

Инструменты дополняют друг друга: DVC фиксирует, что было на входе, MLflow - что получилось на выходе.

## 6. Бизнес-метрики

Технические метрики (F1, Precision, Recall) измеряют качество классификации, но не показывают денежный эффект для банка. Бизнес-метрики оцениваются на лог-данных продуктового использования модели.

### Метрика 1. Снижение ожидаемых финансовых потерь

Базовая величина:

```
Expected Loss = FN * avg_default_loss - FP * avg_lost_revenue
```

Расшифровка элементов:
- `FN` (false negatives) - количество клиентов, для которых модель предсказала "не дефолт", а фактически произошёл дефолт. Это пропущенные дефолтники, банк выдал им кредит и понёс прямые потери
- `avg_default_loss` - средняя сумма потерь на одном дефолтнике в денежных единицах. Считается как `loan_amount * (1 - recovery_rate)`, где `recovery_rate` - доля долга, которую удаётся вернуть после дефолта (типично 0.3-0.5 для беззалоговых потребкредитов)
- `FP` (false positives) - количество клиентов, для которых модель предсказала "дефолт", а фактически клиент платёжеспособен. Банк отказал им в кредите, потерял потенциальную прибыль
- `avg_lost_revenue` - средняя упущенная процентная прибыль на одном ошибочно отклонённом клиенте за срок кредита

Бизнес-эффект модели = `Expected Loss (политика без модели) - Expected Loss (политика с моделью)`. Для A/B-теста рассчитывается отдельно для группы v1 и группы v2, разница интерпретируется как вклад новой версии в финансовый результат.

### Метрика 2. Доля одобрений при фиксированном уровне риска (Approval Rate at Fixed Bad Rate)

Формула:

```
Approval Rate = approved_count / total_applications
при условии bad_rate(approved) <= bad_rate_target
```

Расшифровка:
- `approved_count` - количество клиентов, которым выдан кредит при выбранном пороге `probability < threshold`
- `total_applications` - общее количество поступивших заявок
- `bad_rate(approved)` - доля фактических дефолтов среди одобренных
- `bad_rate_target` - приемлемый для риск-менеджмента уровень плохих среди одобренных, например 5%

Бизнес-эффект модели = прирост доли одобрений по сравнению с консервативной правиловой политикой при том же `bad_rate_target`. Чем точнее модель различает классы, тем большую долю заявок банк может одобрить, не превышая порог риска.

## 7. ONNX-ML (обзор)

ONNX (Open Neural Network Exchange) - кросс-платформенный формат сериализации моделей. Sklearn-модель конвертируется через `skl2onnx.convert_sklearn`, инференс выполняется через `onnxruntime`. Плюсы для production: ускорение инференса в 1.5-3 раза по сравнению с native sklearn, образ уменьшается за счёт исключения scikit-learn и pandas.

## 8. uWSGI и NGINX в production (обзор)

Встроенный сервер Werkzeug, который запускается командой `app.run()`, не предназначен для нагрузочной эксплуатации (один поток, нет защиты от сбоев в воркере). Production-схема Python-бэкенда:

- **uWSGI или Gunicorn** - WSGI application server, держит пул worker-процессов. Сбой воркера не валит весь сервис, перезапуск происходит автоматически
- **NGINX** - reverse proxy перед WSGI: терминирует TLS, отдаёт статику, балансирует запросы между воркерами, ограничивает rate limit, защищает от медленных клиентов

В этом проекте nginx уже подключён в `docker-compose.yml` как reverse proxy перед Flask-приложением, что демонстрирует первую половину связки.
