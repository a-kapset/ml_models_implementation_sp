# План A/B-теста v1 vs v2

Документ описывает дизайн, метрики, статистический анализ и критерий успешности A/B-теста двух версий модели прогнозирования дефолта.

## 1. Цель

Проверить, превосходит ли новая модель v2 текущую продуктивную модель v1 по основной метрике (F1 для класса "дефолт") на реальном трафике сервиса, не ухудшая дополнительную метрику (Recall класса "дефолт") и не увеличивая ожидаемые финансовые потери.

Гипотезы:
- H0: F1(v2) = F1(v1). Версии не отличаются по качеству, наблюдаемая разница случайна
- H1: F1(v2) != F1(v1). Версии отличаются, разница не объясняется случайностью

Тест двусторонний: важно зафиксировать как улучшение, так и деградацию v2.

## 2. Дизайн теста

- **Контрольная группа:** 50% трафика, модель v1 (текущая)
- **Тестовая группа:** 50% трафика, модель v2 (кандидат)
- **Разделение:** на уровне запроса через `random.choice(['v1', 'v2'])` в функции `choose_version` файла `app/model_handler.py`. Включается константой `AB_ENABLED = True` (по умолчанию `False` - все запросы идут на v1). В production-среде переключатель выносится в переменную окружения, чтобы менять режим без пересборки образа
- **Длительность:** определяется размером выборки (см. ниже) и лагом получения метки. Ground truth дефолта известен через 30 дней после прогноза, поэтому даже при достаточном количестве запросов оценка метрик отложена на этот срок

Разделение на уровне запроса достаточно для учебного сценария. В реальном продукте применяется разделение по клиенту (`hash(client_id) mod 2`), чтобы один клиент в течение теста всегда видел одну версию - это исключает дрейф из-за повторных обращений.

## 3. Метрики

### Основная: F1-score для класса "дефолт"

```
F1 = 2 * Precision * Recall / (Precision + Recall)
```

Балансирует точность и полноту:
- Низкий Precision = модель часто маркирует надёжных клиентов как дефолтников, банк отказывает прибыльной клиентуре
- Низкий Recall = модель пропускает реальных дефолтников, банк выдаёт им кредиты и несёт прямые потери

F1 - агрегатная метрика, поэтому к ней применяется bootstrap-доверительный интервал, а не классический z-test (см. раздел 5).

### Дополнительная: Recall для класса "дефолт"

```
Recall = TP / (TP + FN)
```

Расшифровка:
- `TP` (true positives) - дефолтники, которых модель правильно маркировала как дефолтников
- `FN` (false negatives) - дефолтники, которых модель пропустила

Обоснование выбора Recall, а не Precision: финансовая асимметрия ошибок. Пропущенный дефолтник стоит банку всего тела кредита (с учётом recovery_rate потеря порядка 50-70%), ошибочно отклонённый хороший клиент - упущенной маржи на уровне 3-5% от суммы кредита. При прочих равных банк предпочитает перестраховаться, поэтому отдельный контроль Recall критичен даже при положительной динамике F1.

Recall - доля от фиксированной популяции (фактических дефолтников), поэтому к нему применим z-test для двух пропорций.

### Бизнес-метрики

Из `docs/ARCHITECTURE.md`: Expected Loss и Approval Rate at Fixed Bad Rate. Рассчитываются пост-фактум на лог-данных за период A/B-теста, отдельно для группы v1 и группы v2. Используются как итоговый аргумент при принятии решения о переключении на v2.

## 4. Размер выборки и продолжительность

Минимальное число наблюдений в группе для z-теста двух пропорций при заданных параметрах:

```
n_per_group = ((z_alpha/2 + z_beta)^2 * (p1 * (1 - p1) + p2 * (1 - p2))) / (p1 - p2)^2
```

Расшифровка:
- `z_alpha/2` - квантиль стандартного нормального распределения, для уровня значимости alpha = 0.05 (двусторонний тест) равен 1.96
- `z_beta` - квантиль для мощности теста 1 - beta = 0.80, beta = 0.20, равен 0.84
- `p1` - ожидаемое значение Recall в контрольной группе (по результатам обучения v1 ~ 0.36)
- `p2` - значение Recall в тестовой группе при минимально интересном эффекте (MDE = 0.04, p2 = 0.40)

Подстановка: `n_per_group ~ 2400`. Итого нужно ~4800 запросов с известной фактической меткой дефолта. При входящем потоке 200 заявок в день и лаге метки 30 дней реальная длительность теста составляет ~60 дней (24 дня набора + 30 дней ожидания меток + резерв на промежуточные сверки).

## 5. Статистический анализ

### Z-test для Recall

Recall на каждой группе - доля правильно идентифицированных дефолтников среди фактических дефолтников. Это пропорция, к ней применим z-test двух пропорций:

```
z = (p_v2 - p_v1) / sqrt(p_pooled * (1 - p_pooled) * (1/n_v1 + 1/n_v2))
```

где:
- `p_v1`, `p_v2` - наблюдаемые Recall в группах v1 и v2
- `n_v1`, `n_v2` - количество фактических дефолтников в каждой группе (знаменатель Recall)
- `p_pooled = (TP_v1 + TP_v2) / (n_v1 + n_v2)` - объединённая оценка под H0

Решающее правило: если `|z| > 1.96`, отклоняем H0 на уровне alpha = 0.05.

95% доверительный интервал для разности `p_v2 - p_v1`:

```
CI = (p_v2 - p_v1) +- 1.96 * sqrt(p_v1 * (1 - p_v1) / n_v1 + p_v2 * (1 - p_v2) / n_v2)
```

Если CI не содержит ноль - разница статистически значима.

### Bootstrap для F1

F1 - нелинейный функционал, поэтому стандартный z-test к нему не применим напрямую. Используется bootstrap:

1. Из лог-данных каждой группы делается 10000 ресэмплов с возвращением
2. На каждом ресэмпле считается F1
3. 95% CI для F1 группы - перцентили 2.5% и 97.5% распределения
4. Если CI v1 и CI v2 не пересекаются, разница F1 значима

## 6. Критерий успешности

V2 переводится в продуктовую эксплуатацию, если выполнены все три условия одновременно:

1. F1(v2) > F1(v1), 95% bootstrap-CI не пересекаются
2. Recall(v2) >= Recall(v1) - z-test не показывает значимой деградации (нижняя граница 95% CI разности `p_v2 - p_v1` не ниже -0.01)
3. Expected Loss в группе v2 не выше, чем в группе v1, на той же популяции заявок

При невыполнении любого условия v2 откатывается, продуктовой остаётся v1, расследуются причины расхождения метрик в офлайне и на трафике.

## 7. Связь с архитектурой

A/B-маршрутизация реализована внутри монолитного Flask-сервиса, в функции `choose_version` файла `app/model_handler.py`. Решение обусловлено архитектурой:

- Монолит держит обе модели в памяти, выбор версии - это локальный вызов, без сетевого overhead
- Поле `model_version` в JSON-логе каждого запроса позволяет фильтровать события группы v1 и группы v2 для офлайн-анализа в Kibana или ad-hoc через `jq`
- Переключатель `AB_ENABLED` (в production - переменная окружения) управляет включением и отключением теста без передеплоя образа. Можно начать тест, остановить его и вернуться к одной версии без перезапуска

Альтернативные варианты для большего масштаба:
- При числе вариантов больше двух (A/B/C/D) внутреннее `random.choice` неудобно сопровождать. Применяется внешний инструмент управления экспериментами: Unleash, LaunchDarkly, Split.io
- При микросервисной архитектуре маршрутизация выносится на уровень API Gateway или service mesh (Envoy split, Istio VirtualService), сами сервисы не знают о существовании A/B-теста

В обоих случаях монолит и встроенная маршрутизация перестают масштабироваться, но для текущего проекта они оптимальны.

## 8. Практическая проверка

В рамках API A/B-маршрутизация доступна сейчас в двух режимах:

- **Ручной выбор версии** через query-параметр (или поле JSON-body запроса): `POST /predict?version=v1` и `POST /predict?version=v2`
- **Случайное 50/50** включается одной правкой `AB_ENABLED = True` в `app/model_handler.py`. По умолчанию выключено, чтобы штатное поведение сервиса было детерминированным

Примеры curl-запросов с фактическими ответами обоих версий приведены в `README.md`, раздел "API".
